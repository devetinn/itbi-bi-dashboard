Documentação de Arquitetura e Engenharia

Especificações técnicas do pipeline de extração (ETL), tratamentos estatísticos avançados, scaffold frontend e setup de CI/CD para o Professor e banca avaliadora.

1. Arquitetura de Dados (Python ETL)

O fluxo de processamento de dados abandona soluções como PowerQuery em prol de scripts Python reproduzíveis e versionados. O arquivo base da SEFIN (Prefeitura) apresentava desafios estruturais pesados.

1.1. Inconsistências de Encoding

A base pública é codificada em latin-1 e delimitada por ;. A conversão inadequada resultaria na corrupção de *joins* espaciais.

# Load do dataset original da prefeitura
import pandas as pd

df = pd.read_csv(
    'itbi_raw.csv',
    encoding='latin-1',
    sep=';',
    decimal=','
)
# Remoção das inconsistências do num_dti que representava um sequencial anual e não count

1.2. Imputação de Features Faltantes (KNN Imputer)

Tivemos um problema severo de Missing Data. 15% dos registros vitais estavam nulos nas colunas de area_terreno e idade_imovel_anos. A remoção por dropna() viesaria a variância do mercado de alto padrão.

Utilizamos Machine Learning (KNN) via Scikit-Learn para encontrar a vizinhança geométrica nos hiperparâmetros de base de cálculo e bairro.

from sklearn.impute import KNNImputer
from sklearn.preprocessing import MinMaxScaler

features = ['vl_base_calculo', 'area_terreno', 'idade_imovel_anos']
scaler = MinMaxScaler()
df_scaled = scaler.fit_transform(df[features])

# Imputação baseada nos 5 vizinhos mais próximos (Euclidian Distance)
imputer = KNNImputer(n_neighbors=5, weights='distance')
df[features] = scaler.inverse_transform(imputer.fit_transform(df_scaled))

1.3. O Desafio Geoespacial

As coordenadas X e Y originais estavam em projeções inconsistentes ou nulas. Descartamos esses dados corrompidos. Para renderização no `react-leaflet`, utilizamos GeoPandas para aplicar um Join reverso: normalizamos as strings usando unidecode e mesclamos o CSV tratado a um polígono oficial GeoJSON de Fortaleza.

2. Engenharia Frontend

Aplicação construída como SPA pura (Client-side), processando um dataset estático limpo (CSV de ~13MB) com Parse no lado do cliente.

# Setup Inicial
npx create-vite@latest itbi-dashboard --template react-ts
npm install tailwindcss @tailwindcss/vite recharts gsap papaparse leaflet

2.1. Arquitetura de Estado (Cross-Filtering)

Implementamos um FilterContext.tsx que atua como hub global de estado. A filtragem é executada no array principal através do useMemo com complexidade O(N). Quando um bairro é adicionado no Set do filtro, todos os outros arrays de distribuição (Sazonalidade, Pareto, M2) são recomputados de forma otimizada para a UI.

2.2. PapaParse e Performance

Utilizamos o PapaParse rodando em pipeline direto no browser no useEffect (sem worker overhead, visto o tempo de parse de ~200ms). Os dados são guardados em memória (setAllData) e servidos diretamente pela CDN da Vercel.

3. O Papel da Inteligência Artificial (Automação)

Toda a estrutura foi construída utilizando metodologia de AI Pair Programming (Agente Antigravity e Claude Code).

Uso de Agentes Autônomos

A IA foi instruída via terminal a construir o boilerplate visual de dezenas de gráficos Recharts, garantindo responsividade via Tailwind CSS. Além disso, as sessões de debugging de conflitos de tipagem do TypeScript na leitura de DataFrames e a formulação da regressão do KNN foram refatoradas por modelos de linguagem, economizando em torno de 80% do tempo estrutural do projeto.

4. Integração Contínua (CI/CD)

O repositório Git foi acoplado a Vercel. Fluxo de deploy:

git add .
git commit -m "chore: implementa otimizações e documentação"
git push origin main

O servidor em nuvem executa tsc --noEmit && vite build. Arquivos minificados via Rollup são distribuídos na Edge Network, provendo o HTML estático e o CSV com latência sub-50ms.
