A Sequência Lógica do Projeto

O Ciclo de Vida dos Dados Imobiliários

Acompanhe, passo a passo, como capturamos a base oficial do governo, tratamos os registros e estruturamos as informações para a criação deste painel analítico.

01

A Origem e Diagnóstico do Dataset

A base de dados oficial foi extraída do Portal de Dados Abertos da SEFIN (Fortaleza). Recebemos um arquivo CSV denso, com a codificação legada 'latin-1' e separadores complexos.

Registros Iniciais
94.970
Colunas Brutas
37 colunas

O diagnóstico revelou problemas graves: a coluna TIPO_TERRENO estava 100% nula. Pior ainda, quase 88% dos imóveis não possuíam a data exata da transação (data_transacao_itbi) preenchida pela prefeitura.

Foco de Análise
Conhecer o estado real do dado bruto é vital. Não podíamos confiar na "data da transação", então nossa estratégia precisou migrar para a "data de cadastramento" e "ano do débito" para garantir precisão temporal.
02

O Processo de ETL (Limpeza e Padronização)

Para não trabalhar com informações erradas, rodamos um script Python para realizar o processo de ETL (Extração, Transformação e Carga). As mudanças mais importantes que fizemos para dar precisão à análise foram:

  • Excluímos colunas inutilizáveis (100% nulas).
  • Convertemos a acentuação para o formato padrão (UTF-8), salvando nomes vitais como "Passaré" e "Cocó".
  • Criação de Novas Colunas: Calculamos atributos estratégicos que o governo não entregou prontos, gerando o vl_m2_base (Valor por Metro Quadrado), a categoria_uso (isolando Residenciais vs Comerciais) e criando as faixas_de_valor (ex: Acima de 1 Milhão).
Foco de Análise
A padronização permite que o painel grupe os dados de forma confiável. Sem essas transformações, o sistema leria o bairro "Aldeota" de três formas diferentes, dividindo incorretamente as métricas.
03

A Recuperação de Dados Faltantes (KNN)

Havia um buraco perigoso na amostra: mais de 10.000 imóveis não tinham a idade anotada, e quase 6.890 transações não possuíam o Valor Base de Cálculo definido.

Ao invés de descartar e perder a base histórica de anos inteiros, aplicamos o algoritmo KNN (K-Nearest Neighbors). O método buscou, para cada imóvel em branco, os vizinhos matematicamente mais próximos (mesmo bairro, mesmo padrão de construção) para projetar com exatidão o valor ou área ausentes.

Taxa de Cobertura Final de Valores
92.7% dos registros salvos
Foco de Análise
Salvar essas informações via Machine Learning garantiu a representatividade financeira (VGV) no mapa sem comprometer o rigor estatístico da base da Prefeitura.
04

Conexão do Sistema Front-end (React)

Com o arquivo final gerado (itbi_clean.csv), desenhamos a arquitetura do painel visual. Como e de onde o sistema puxa os dados?

  • Leitura Cliente-Lado: O sistema NÃO usa um banco de dados relacional clássico (como SQL). Nós inserimos o CSV compactado na nuvem da Vercel. Quando a página carrega, ela lê e analisa toda a planilha na hora usando a biblioteca PapaParse.
  • Filtros Interligados: Como o dado já está "limpo" no navegador, nossos gráficos conversam entre si quase sem atraso (delay nulo).
  • Mapa Georeferenciado: Ignoramos os problemas de coordenadas falhas originais e conectamos o nome do Bairro a um polígono oficial de Fortaleza (GeoJSON), pintando as áreas exatas.
Foco de Análise
Esta arquitetura reduz custos de infraestrutura a zero e cria a interatividade ágil necessária para os "Filtros Cruzados", onde a diretoria pode isolar um segmento de luxo com um único clique.
