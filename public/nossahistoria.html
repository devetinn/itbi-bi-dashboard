Processo.

A história completa de como o Dashboard ITBI Fortaleza foi construído — da ideia ao deploy, passando pela coleta de dados reais, ETL em Python, design system e cada decisão técnica ao longo do caminho.

Capítulo 1

O ponto de partida

Um projeto acadêmico com um requisito incomum: rastreabilidade completa. O professor não queria Power Query nem transformações invisíveis — queria ver o dado bruto da prefeitura virar insight de ponta a ponta.

📋
O desafio proposto
Coletar dados abertos de ITBI do Portal da Prefeitura de Fortaleza, processar via script Python documentado e construir um dashboard web interativo que permita análise temporal, espacial e de mercado — tudo publicado e acessível.
🏛️
Fonte: Dados Abertos
Portal dados.fortaleza.ce.gov.br — dataset público da SEFIN (Secretaria Municipal de Finanças). CSV bruto com todos os lançamentos de ITBI desde 1996.
🔒
Regra do professor
Nenhuma transformação em ferramentas como Excel ou Power Query. O ETL precisa ser em código versionado, com cada etapa documentada e reproduzível.

Capítulo 3

O pipeline ETL em Python

O etl_itbi.py transforma o CSV bruto da prefeitura em dados limpos, padronizados e prontos para o dashboard — sem depender de nenhuma ferramenta externa.

E
Extract
Leitura do CSV bruto
O arquivo da SEFIN usa encoding latin-1 (não UTF-8) e separador ponto-e-vírgula. Ler com as configurações erradas destroça nomes de bairros com acentos.
import pandas as pd # encoding correto: latin-1, separador: ; df = pd.read_csv( 'itbi_raw.csv', encoding='latin-1', sep=';', decimal=',' )
T
Transform
Limpeza e campos derivados
Remoção de duplicatas, conversão de datas, normalização de nomes de bairros, cálculo de campos que o CSV original não tinha: idade do imóvel, valor por m² e faixa de valor.
# Campos derivados df['idade_imovel'] = ano_atual - df['ano_construcao'] df['valor_m2'] = df['vl_base_calculo'] / df['area_terreno'] df['faixa_valor'] = pd.cut( df['vl_base_calculo'], bins=[0, 100e3, 300e3, 500e3, 1e6, float('inf')], labels=['<100k','100-300k','300-500k','500k-1M','>1M'] )
L
Load
Saída em UTF-8 para o browser
O CSV final (itbi_clean.csv) sai em UTF-8, pronto para ser lido pelo PapaParse diretamente no browser — sem backend, sem banco de dados.
# Salva limpo — UTF-8, sem BOM df.to_csv( 'public/itbi_clean.csv', index=False, encoding='utf-8' ) # Ground truth para validação no frontend print(f"Total: {len(df):,} registros") print(f"Soma vl_base: R${df.vl_base_calculo.sum():,.2f}")

O Design System PMF

Antes de escrever um componente, o design system foi definido. Todos os tokens ficam em src/index.css como variáveis CSS nativas — sem dependência de tema JS, sem runtime overhead.

Paleta de cores — 4 cores primárias + escalas

--c-teal #009889 Positivo / sucesso / crescimento
--c-petrol #325565 Estrutural — sidebar, cabeçalhos, navs

--c-bg #FAFAF7 Fundo global — tom creme quente, não branco puro
Tipografia
Archivo para títulos e labels (bold, display).
Space Grotesk para corpo de texto.
JetBrains Mono para números, código e valores.
Elevação & Animação
3 níveis de sombra (--sh-1, --sh-2, --sh-3). Transições em 4 velocidades (--t-fast 150ms até --t-bounce 600ms). Framer Motion para todas as entradas de página.

Por que sem backend?
O dataset completo tem 94.970 linhas — 13 MB de CSV. O PapaParse lê isso em ~200ms no browser moderno. Sem servidor significa zero latência de API, deploy gratuito e disponibilidade garantida pela CDN do Vercel. Para um dataset de análise histórica que não muda a cada minuto, é a escolha certa.

Capítulo 7

Do código ao ar

O deploy é automático — cada git push origin main dispara um novo build no Vercel em menos de 2 minutos.

1
Repositório GitHub
O código vive em github.com/devetinn/itbi-bi-dashboard. Cada branch tem preview deploy automático no Vercel.
2
Push → Build
O Vercel detecta o push, executa npm run build (tsc + vite build) e valida a saída. Build típico: ~1.1s.
3
CDN global
Os assets são distribuídos em edge nodes globais. O CSV de 13MB é servido com cache, carregando em ~200ms para usuários no Brasil.
4
Zero config necessária
O vercel.json tem apenas 3 linhas: buildCommand, outputDirectory e framework. O resto é inferido automaticamente.
{ "buildCommand": "npm run build", "outputDirectory": "dist", "framework": "vite" }
Capítulo 8

Decisões e aprendizados

As escolhas que pareciam simples e as armadilhas que não eram.

Perguntas frequentes do processo
Por que PapaParse no browser e não SQLite?
O CSV tem ~13 MB, e o usuário filtrado médio consulta menos de 5% dos registros. O PapaParse parseia tudo em ~200ms no browser moderno. SQLite via WASM seria mais poderoso para queries complexas, mas adicionaria 300 KB de bundle e complexidade sem benefício perceptível para este caso de uso.
Por que Tailwind v4 junto com variáveis CSS customizadas?
Tailwind cuida do layout (flex, grid, spacing utilitário, responsivo) — onde ele é imbatível em velocidade de desenvolvimento. As variáveis CSS cuidam do design system (cores, tipografia, sombras) — onde precisamos de semântica e não de nomes utilitários. Os dois coexistem perfeitamente.
Por que Recharts em vez de Chart.js ou D3 direto?
Recharts é composable em JSX — você monta o gráfico como componentes React. Chart.js exige manipulação de refs e configuração imperativa (mais verboso no React). D3 direto dá mais controle, mas exige gerenciar o DOM manualmente — perigoso misturar com o virtual DOM do React. Para esse nível de complexidade, Recharts é o meio-termo certo.
GSAP para a intro e Framer Motion para o resto — por quê dois?
GSAP é o melhor do mercado para animações de tela cheia e sequências complexas com timeline. A intro era um caso ideal: partículas flutuando, logo revelando, contador. Framer Motion é mais ergonômico para animações de componentes React (variants, AnimatePresence, layout animations) — onde o GSAP seria mais verboso.
O que eu faria diferente?
Usaria streaming de CSV no PapaParse desde o início — parsear chunk por chunk e atualizar a UI progressivamente, em vez de esperar o arquivo completo. Também teria criado os tokens de design system antes do primeiro componente, em vez de refatorar cores hardcoded depois.
