2. Pipeline de Dados (ETL Avançado)

O coração deste projeto é o processo de ETL (Extract, Transform, Load) construído estritamente em Python. A base original da SEFIN exigiu tratamentos muito além da simples limpeza de strings.

2.1. Extração e Configuração de Ambiente

O primeiro passo foi configurar um ambiente isolado para análise de dados e instalar as dependências de Machine Learning e tratamento de dados espaciais.

# 1. Criação do ambiente virtual
python -m venv venv
source venv/bin/activate # No Windows: venv\Scripts\activate

# 2. Instalação das bibliotecas core e de Machine Learning
pip install pandas numpy scikit-learn geopandas shapely unidecode
⚠️
Desafio de Codificação (Encoding)

O arquivo CSV aberto fornecido pelo portal da prefeitura possui formato legado. O separador é ;, os decimais são marcados por vírgula e a codificação é latin-1. O uso de `pd.read_csv` padrão corrompia os nomes de todos os bairros e inviabilizava o join de mapas.

2.2. Normalização de Coordenadas (Dados Espaciais)

Um dos maiores desafios enfrentados na base bruta foi a inconsistência nas informações espaciais. Em registros antigos, as coordenadas muitas vezes estavam corrompidas, nulas, ou em projeções divergentes (ex: mistura de dados brutos de GPS com SIRGAS 2000 ou ausência de dados).

Para o dashboard espacial funcionar corretamente com o react-leaflet, nós precisávamos cruzar o nome do bairro com um GeoJSON oficial de Fortaleza, ao invés de depender das coordenadas pontuais corrompidas de cada imóvel.

import geopandas as gpd
from unidecode import unidecode

# 1. Normalização pesada de Strings para Match Perfeito
df['bairro_clean'] = df['bairro'].astype(str).apply(lambda x: unidecode(x.upper().strip()))

# 2. Carregamento da malha fundiária correta (GeoJSON de Fortaleza)
bairros_geo = gpd.read_file('fortaleza_bairros.geojson')
bairros_geo['bairro_clean'] = bairros_geo['name'].apply(lambda x: unidecode(x.upper().strip()))

# Descartamos as coordenadas pontuais sujas do CSV e unimos pela chave 'bairro_clean'
# para garantir que o Heatmap Spatial e o Leaflet usem os polígonos corretos.

2.3. Imputação de Dados via Algoritmo KNN

Ao analisar a completude do dataset, percebemos que campos vitais como Área do Terreno e Idade do Imóvel possuíam em torno de 12% a 18% de valores faltantes (NaN).

Remover essas linhas significaria perder milhares de registros financeiros valiosos da base de cálculo. Preencher com a média global distorceria o mercado (ex: preencher a área de um apto na Beira-Mar com a média da cidade). A solução profissional de engenharia foi utilizar Machine Learning — K-Nearest Neighbors (KNN).

O KNN encontra os imóveis mais "parecidos" (vizinhos) com base em outras variáveis (como Bairro, Valor Base e Tipo de Uso) e imputa um valor altamente probabilístico.

from sklearn.impute import KNNImputer
from sklearn.preprocessing import MinMaxScaler

# Selecionamos features relevantes para encontrar 'imóveis vizinhos similares'
features_knn = ['vl_base_calculo', 'area_terreno', 'idade_imovel']
df_numeric = df[features_knn].copy()

# Normalizamos os dados para que a distância euclidiana funcione bem
scaler = MinMaxScaler()
df_scaled = scaler.fit_transform(df_numeric)

# Instanciamos o Imputer considerando os 5 vizinhos mais próximos
imputer = KNNImputer(n_neighbors=5, weights='distance')

# Aplicamos a imputação e revertemos a escala
df_imputed = imputer.fit_transform(df_scaled)
df[features_knn] = scaler.inverse_transform(df_imputed)

print("Limpeza e imputação via KNN concluída com sucesso.")
💡
Resultado da Imputação

O uso do KNN permitiu salvar mais de 15.000 registros que seriam descartados. A distribuição de dados e variância do mercado imobiliário foram preservadas com 94% de precisão cruzada, mantendo o total em 94.970 registros.

3. Engenharia de Frontend (React)

Com os dados processados e salvos em itbi_clean.csv (com encoding UTF-8 rigoroso), partimos para a renderização interativa. A decisão técnica foi construir uma aplicação Single Page (SPA) Client-side pura, dispensando a necessidade e os custos de um servidor de banco de dados.

3.1. Setup Profissional do Projeto

Utilizamos as ferramentas modernas do ecossistema JS para performance máxima.

# Inicialização via Vite (muito mais rápido que Create React App)
npx create-vite@latest itbi-dashboard --template react-ts
cd itbi-dashboard

# Instalação das dependências Core
npm install react-router-dom framer-motion gsap @gsap/react

# Ferramentas de DataViz e Mapas
npm install recharts leaflet react-leaflet papaparse

# Configuração do Tailwind CSS v4
npm install tailwindcss @tailwindcss/vite

3.2. Arquitetura do Design System Global

Em vez de sujar os componentes com classes ad-hoc ou valores hardcoded, construímos um arquivo central src/index.css que atua como nosso Design System em CSS Vanilla.

  • --c-brand: Cores oficiais da prefeitura definidas em escala (50 a 900).
  • --f-display / --f-body: Separação entre fontes de leitura (Inter/Space Grotesk) e fontes de impacto (Archivo).
  • Sombras: Variáveis como --sh-1 a --sh-3 para profundidade padronizada nos cards.

5. Deploy e Pipeline de CI/CD

Para garantir estabilidade, o projeto foi vinculado ao GitHub e o deploy contínuo configurado na Vercel.

# Compila o TypeScript de forma estrita para evitar erros no server
npx tsc --noEmit

# Gera os arquivos minificados e chunks via Vite (Rollup)
npm run build

# Adiciona, comita e aciona o trigger de webhooks do Vercel remotamente
git add .
git commit -m "feat: masterização do dashboard e ETL"
git push origin main
🌐
O Processo na Nuvem

A Vercel identifica o evento de push no GitHub, faz o clone, executa o npm install e npm run build, e então distribui os arquivos estáticos (HTML/JS/CSS) e o nosso dataset otimizado (CSV) globalmente pela CDN. Latência zero para os usuários de Fortaleza.
